GOAL
Prove that an AUTHORITATIVE-ROOM + SERIALIZED-OP model holds plan's zero-data-loss guarantee on plan's ORDERED, INTERDEPENDENT, VERSIONED document, on top of the suite's proven auth and durable-persistence plumbing. Stand up the thinnest server that: authenticates a ws upgrade via @suite/auth-client (company on the session), holds one authoritative plan document per room in memory, receives OPERATIONS (not values) over ws, applies them in arrival order through plan's EXISTING reducers and validatePlan, commits the result transactionally (retro's discipline) BEFORE acking/broadcasting, and rebroadcasts. Then drive the hard collisions from two concurrent clients and assert the document never lands in a state that fails validatePlan and never silently loses an edit in a way that breaches the bar.

THE THREE DECISIONS BAKED IN (director-ruled this session; do not reopen in PROPOSE)
- ACCESS = BLEND. A room carries a policy: OPEN-LINK (anyone with the shareToken joins; identity is SELF-ASSERTED, a claimed display name, not authenticated) or COMPANY-ACCOUNTS-ONLY (every participant must be an authed member of the room's company). The manager (authed, entitled, a company member) creates the room and picks the mode; the company name is always visible ("IBM - Data plan Qtr 3"). An open-link room is only as private as its link; a claimed name is never the company-membership guarantee. For the spike, exercise BOTH modes at the upgrade boundary; the collision proof itself is identity-agnostic.
- FACILITATOR HANDOFF = SILENT REASSIGNMENT (poker). Reuse poker's assignFacilitator semantics: on facilitator leave, silently promote the next ELIGIBLE participant. Eligible = an authed company member. RULING that falls out of poker's own code: when NO eligible member remains (e.g. an open-link room whose only authed user, the manager, dropped), facilitatorId goes null and facilitator-gated actions are UNAVAILABLE (a freeze) until an authed company member (re)joins. This is poker's actual null-return behaviour, not a new invention. For the spike, the only facilitator-gated action is SESSION LIFECYCLE (end session); structural plan mutation is open to all participants ON PURPOSE, to maximise the conflict surface the spike must survive. (Editing-rights modes are a post-spike feature, parked.)
- CHARGE-LATER = OPTION PRESERVED, NOTHING GATED YET. Register plan as a first-class hub-entitled app (APP_DOMAIN, ALLOWED_APP_DOMAINS, api-key config, an entitlement row), keep every room and every write COMPANY-SCOPED and COUNTABLE, and leave a consume() seam in place but UNUSED (granted unlimited / never called), exactly as RAID shows the pattern. Build NO billing, NO Stripe, NO tiers UI, NO paywall. A future paywall is then a config value plus one consume() call, not a re-architecture. (This is the deliberate inverse of Brief 10 R5/R6, consistent with the reversal above.)

MODEL ON RETRO, NOT POKER (the structural correction from the feasibility doc)
"poker's ws + retro's SQLite" is a false split: retro is ALREADY ws + durable SQLite + company-scope in one app, and poker's ws layer is the inferior, ephemeral, in-memory one. BUILD THE SPIKE AS A SMALL RETRO: a ws layer over a better-sqlite3 store that commits per-mutation inside db.transaction BEFORE broadcasting, company-scoped at the upgrade. Borrow from poker only the ws-upgrade + shareToken + silent-reassignment shapes. CRITICAL: both siblings are LAST-WRITE-WINS and neither solves conflict; the conflict model is NET-NEW and is the entire point of this spike. Do not copy either sibling's write model; copy retro's DURABILITY DISCIPLINE and MIGRATION IDIOM only.

THE LOAD-BEARING SEAM (section 5.1 of the feasibility doc, ruled in)
The client-to-server contract is OPERATIONS, not values. The server:
1. receives an op over ws (an op is one of plan's existing actions, or a tightly-mapped subset; see PROPOSE),
2. applies it to the authoritative in-memory plan document IN ARRIVAL ORDER (Node's single thread serialises this for free),
3. validates the result with plan's EXISTING reducers and validatePlan, and commits the resulting document TRANSACTIONALLY (retro's per-mutation db.transaction) BEFORE acking or broadcasting,
4. rebroadcasts the applied op (or the new authoritative document) to the room; each client's store applies it.
validatePlan stays the single load/guard spine it already is, now on the SERVER side of the boundary. The autosave envelope (sprintplan:board) remains the CLIENT-side local backstop and is unchanged by the spike.

RULINGS BAKED INTO THIS BRIEF (director-ruled R1 to R7, do not silently change)
- R1 OPERATIONS OVER VALUES. The wire carries ops, never "here is the new field value". An op names an intent the server re-applies through plan's reducers. This is what makes arrival-order serialisation a real guarantee rather than poker's silent overwrite. (Feasibility 5.1.)
- R2 THE SERVER IS AUTHORITATIVE; IT VALIDATES, THEN COMMITS, THEN BROADCASTS. No op is acked or broadcast until validatePlan passes on the resulting document AND the document is committed inside a transaction. A rejected op is surfaced to its sender (see PROPOSE on how), never silently dropped. (Retro's discipline plus plan's validatePlan spine.)
- R3 THE DATA-LOSS BAR CROSSES THE NETWORK INTACT. The zero-data-loss quality bar now spans a socket. The spike's PASS/FAIL turns on it: concurrent ops must NEVER leave a document that fails validatePlan on reload, and must never silently lose an edit in a way that breaches the bar. The client-side sprintplan:board autosave envelope stays the local backstop and is untouched.
- R4 THE DEP-PRUNE INVARIANT SURVIVES CONCURRENCY (the project's hard rule, now multiplayer). plan's standing ruling (deleting a story or epic must atomically prune every dep pair referencing the deleted ids) must hold UNDER A RACE: a delete op and a dep-add op naming the same story, applied in EITHER order, must converge to a document that passes validatePlan (no dangling dep that fails the next load). This is the single most important collision in the proof and the direct multiplayer form of the existing invariant.
- R5 ACCESS IS BLEND; HANDOFF IS SILENT-REASSIGN-THEN-FREEZE. The three baked decisions above are constraints, not PROPOSE items. The company name is always visible; an open-link claimed identity is never a membership guarantee.
- R6 MODEL ON RETRO. ws + better-sqlite3 + per-mutation transactional commit + meta/schema_version migration idiom + company-scoped upgrade. Poker is secondary reference only. Neither sibling's LWW write model is copied.
- R7 OPTIONALITY WITHOUT GATING. Company-scoped, countable persistence + a registered hub entitlement + an UNUSED consume() seam. No billing, no paywall, no tiers UI in this spike or its follow-on until a separate director ruling pulls the trigger.

WHAT THE SPIKE BUILDS (thinnest vertical, throwaway-tolerant)
- A standalone Node service (its own process/port, like the siblings) with a ws server whose UPGRADE is authenticated by @suite/auth-client verifySession(cookie), reading {userId, entitled, company}; rooms keyed company-scoped (poker's ${companyId}-${room}); a shareToken for join; the blend policy enforced at join.
- An authoritative in-memory plan document per room, seeded from a minimal valid plan, persisted via better-sqlite3 committed per applied op inside db.transaction BEFORE broadcast.
- An OP handler that re-uses plan's EXISTING reducers and validatePlan to apply and guard each op. NO new domain maths; the existing action set is the op vocabulary (see PROPOSE on granularity/versioning).
- A minimal two-client harness (Playwright multi-context, the shape of poker's multi-user-room.spec.js) that drives the collision cases below. The board UI is NOT in scope; drive ops directly or through the thinnest possible client shim.

ASSERTED CASES (the PASS/FAIL of the spike; assert exact outcomes against real plan states)
Drive each from TWO concurrent clients against one room, then RELOAD the persisted document and run validatePlan:
1. SAME-STORY CONCURRENT MOVE. Both clients move the same story to different sprints in the same window. Result: exactly one coherent placement, no orphaned or duplicated story, validatePlan passes, both clients converge to the same state.
2. DEP-ADD vs BLOCKER-DELETE (R4, the headline). Client A adds a dependency naming story S; client B deletes S, in the same window, in BOTH orders. Result in both orders: a document that passes validatePlan with NO dangling dep referencing a deleted id. (The atomic-prune invariant holding under a race.)
3. CONCURRENT REORDER WITHIN A SPRINT. Both clients reorder stories in the same sprint at once. Result: a single converged order, no lost or duplicated story, validatePlan clean. (Position-sensitive ops are the subtlest; this case decides whether arrival-order alone suffices or a reorder needs an explicit rule, see PROPOSE.)
4. CRASH MID-FLIGHT. Kill the service immediately after an op is acked and before the next; on restart the persisted document reflects exactly the acked ops and passes validatePlan (retro's per-mutation transactional commit means worst-case loss is one in-flight, un-acked op).
5. STALE-VERSION OP. An op arrives carrying a base version older than the authoritative document (if versioned ops are adopted, see PROPOSE). Result: rejected and surfaced to its sender, not silently applied over newer state.
6. REJECTED OP IS VISIBLE. Any op that would make validatePlan fail is rejected and the sender is told; the authoritative document and all other clients are unaffected. No silent drop.
7. AUTH + COMPANY SCOPE AT THE BOUNDARY. A ws upgrade without a valid session is refused; an op or join carrying a different company's room id is refused (404-class, never a cross-company leak), matching the suite rule. Both blend modes are exercised: open-link join via shareToken with a self-asserted name, and company-accounts-only join requiring an authed company member.

PASS = all seven hold. FAIL = any case produces a document that fails validatePlan on reload, a silent edit loss that breaches the bar, a cross-company leak, or client divergence that arrival-order plus the chosen conflict rules cannot reconcile. A FAIL is a SUCCESSFUL spike: it tells us the model is wrong before we build on it. Write the finding either way.
